Status Legend

Every feature and claim in this document is tagged with one of four status indicators. Use these to quickly understand what is production-ready, what has known gaps, what has been actively repaired, and what is on the future roadmap.

BUILT
Fully implemented. Code exists, tests cover the happy path, and the feature is accessible through the documented UI or API.
PARTIAL
Implemented with known gaps. Core logic exists but edge cases, integrations, or UI polish remain incomplete. Documented limitations included.
FIXED
A weakness was identified during review and has been resolved. See Section 8 for full problem/solution/file details.
ASPIRATIONAL
Phase 3 target — not yet built. Claimed in pitch materials or roadmap but no corresponding code exists in the current monorepo.

S5 — "One Trip Digitized" : 6-Step Lifecycle Detail

Step 1 — Passenger Opens App

(passenger)/index.tsx — fetches balance via GET /smartcard/balance, renders active buses via GET /buses/active (Socket.io fallback for live updates).

Step 2 — Driver Starts Shift

(driver)/index.tsx — pre-trip checklist gated before POST /trips/start. Bus ID and route selected from dropdown. Trip record created in PostgreSQL, BullMQ job enqueued.

Step 3 — Passenger Boards & Taps SmartCard

(driver)/validate.tsx renders tap target UI. Card tap triggers POST /smartcard/tap → smartcard.service.ts tapCard() — atomic Prisma transaction: debit balance, write Transaction record, emit FARE_TAP Socket.io event, enqueue fraud job.

Step 4 — Live GPS Tracking

Driver app emits location every 5 s via socket/gps.gateway.ts. Backend workers/gps.worker.ts batches 50 pings, flushes every 5 s to GpsEvent table. Passenger (passenger)/tracking.tsx receives GPS_UPDATE and moves MapView marker in real time.

Step 5 — ETA Calculation

services/eta.service.ts — haversine distance from current bus location to each remaining stop, divided by average segment speed. Returns ETA + confidence score (0–1). Displayed in (passenger)/tracking.tsx route pill.

Step 6 — Trip End & Analytics Write

Driver taps "End Trip" → POST /trips/end → trip.service.ts writes trip summary (duration, fare revenue, stop count) to PostgreSQL. analytics.service.ts aggregates on next cron cycle. Admin fleet.tsx KPIs update on next poll.

Section 8

Four Weak Points Found & Fixed

During code review, four implementation weaknesses were identified where pitch claims were not backed by working code. Each fix is documented with the exact problem, the solution implemented, and the files changed.

✓
FIX 1 — Push Notification Worker Was Missing
FIXED Notifications silently dropped — Expo Push API never called
Problem
notification.service.ts correctly queued push notification jobs to the BullMQ notifications queue. However, no worker was registered to consume this queue. Jobs piled up unprocessed and notifications were silently discarded. Passengers and drivers never received any push alerts.
Solution
Created workers/notification.worker.ts that registers on the notifications queue. For each job, it: (1) resolves the target audience — if route-scoped, queries Transaction table for users who boarded that route in the last 30 days; (2) fetches their Expo push tokens from UserDevice table; (3) splits into 100-token batches to respect Expo API limits; (4) POSTs each batch to https://exp.host/--/api/v2/push/send; (5) implements exponential backoff on 429 responses; (6) logs push receipts. Worker registered in server.ts startup sequence.
Files
apps/backend/src/workers/notification.worker.ts (new file)
apps/backend/src/server.ts (register worker on startup)
✓
FIX 2 — Revenue LineChart Was a Dead Import
FIXED Admin revenue page imported Recharts but rendered nothing
Problem
admin/src/pages/revenue.tsx had import {'{'} LineChart, Line, XAxis, YAxis {'}'} from 'recharts' at the top of the file but the JSX returned only static summary cards — the LineChart component was imported but never placed in the render tree. The pitch slide S8 shows a revenue trend chart. Investors and JUTC staff saw no chart.
Solution
Added GET /admin/revenue/daily endpoint to the backend returning [{'{'} date: string, totalJMD: number {'}'}] bucketed by calendar day (Prisma groupBy on Transaction.createdAt truncated to day). Updated revenue.tsx to fetch this endpoint on mount and render a <LineChart> with date on X-axis and totalJMD on Y-axis. Added date range selector (7d/30d/90d) that refetches with query param.
Files
apps/backend/src/services/analytics.service.ts — added getRevenueDailySeries(days)
apps/backend/src/controllers/admin.controller.ts — added handler
apps/backend/src/routes/admin.routes.ts — added route
apps/admin/src/pages/revenue.tsx — connected chart
✓
FIX 3 — Demand Heatmap Had No Stop Dimension
FIXED getDemandHeatmap() returned only hourly totals — no spatial insight
Problem
analytics.service.ts getDemandHeatmap() returned {'{'} counts: number[] {'}'} — a 24-element array of hourly tap totals. The admin demand page rendered a bar chart of this data. The pitch slide S8 claims "Demand Intelligence" with spatial stop-level heatmap capability. There was no per-stop breakdown anywhere in the codebase — operators could not identify which stops were chronically overcrowded.
Solution
Extended getDemandHeatmap(routeId?, date?) to return {'{'} byHour: number[24], byStop: {'{'} stopId: string, name: string, count: number {'}'}[], total: number {'}'}. The byStop array is produced by a Prisma groupBy on Transaction.stopId joined with Stop.name, ordered descending by count. Admin demand.tsx now renders two charts side-by-side: the existing hourly BarChart plus a new horizontal BarChart of per-stop demand, and a sortable stop table below.
Files
apps/backend/src/services/analytics.service.ts — updated getDemandHeatmap() return type
apps/admin/src/pages/demand.tsx — added per-stop horizontal bar chart
✓
FIX 4 — Validate Screen Had No Offline Fallback
FIXED Pitch claims offline-first but validate.tsx failed silently with no network
Problem
(driver)/validate.tsx called api.post('/smartcard/tap', payload) directly. If the device had no network connection (common in rural Jamaica routes), the tap failed silently — the status UI showed an error, the passenger was not charged, and no record was created. The pitch explicitly claims "offline-first" capability. There was no queue, no retry, and no manual override for network failure.
Solution
Added "Network Offline" as a first-class override reason in the manual override bottom sheet. When selected, the driver confirms the passenger has boarded, and the app calls POST /smartcard/tap with txType: OVERRIDE and overrideReason: 'NETWORK_OFFLINE' — this still requires network for the override record itself but is explicitly documented as the protocol. The Transaction record is written with fraudFlagged: false and a reconciliation flag so finance can audit cash-equivalent boardings. The full offline queue approach (local SQLite + sync) is tracked as a Phase 3 enhancement.
Files
apps/mobile/app/(driver)/validate.tsx — added Network Offline override reason
apps/backend/src/services/smartcard.service.ts — handle OVERRIDE txType with reconciliation flag

Known Issues Not Yet Fixed

SMS Alerts Not Integrated — notification.service.ts only delivers push notifications via Expo Push API. SMS via Twilio or a Caribbean SMS gateway is referenced in the pitch but not implemented. Passengers without the app installed receive no service alerts. Tracked as Phase 2 target.
TimescaleDB and BigQuery Not Integrated — Slide S14 mentions TimescaleDB for time-series GPS data and BigQuery for data warehouse analytics. The current deployment uses vanilla PostgreSQL 14 with no time-series extension activated and no BigQuery pipeline. All analytics run as Prisma aggregate queries against PostgreSQL, which will not scale past ~500k daily transactions without optimisation. Tracked as Phase 3 target.

Required Environment Variables

DATABASE_URL
PostgreSQL connection string. e.g. postgresql://postgres:password@localhost:5432/jutc_db
REDIS_URL
Redis connection string. e.g. redis://localhost:6379
JWT_SECRET
Secret for signing access tokens. Min 32 characters. Generate with openssl rand -hex 32
JWT_REFRESH_SECRET
Separate secret for refresh tokens. Must differ from JWT_SECRET.
QR_SECRET
Secret for signing SmartCard QR JWTs. Scoped separately to limit blast radius if leaked.
PORT
Backend HTTP port. Default 3000.
